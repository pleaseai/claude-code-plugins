# ARCHITECTURE.md

> Bird's-eye view of the claude-code-plugins repository.
> Last updated: 2026-03-17

## Overview

This repository is a **curated marketplace of plugins for Claude Code**, maintained by Passion Factory. It aggregates plugins from multiple upstream sources, generates and syncs skill artifacts, and serves a web-based marketplace frontend. The system manages ~35 plugins across 4 distinct plugin types, each with its own ingestion pipeline.

The core workflow: upstream skill/extension repositories are registered in `scripts/meta.ts`, pulled as git submodules, and the `scripts/cli.ts` tool generates standardized Claude Code plugin artifacts into `plugins/`. Users install plugins via the Claude Code CLI marketplace commands.

## Entry Points

| Entry Point | Purpose |
|---|---|
| `scripts/cli.ts` | Main CLI tool — `init`, `sync`, `check`, `cleanup` commands for managing plugin sources and artifacts |
| `scripts/meta.ts` | Registry of all plugin sources — Type 1 submodules, Type 2 vendors, Type 3 manual, Type 4 extensions |
| `.claude-plugin/marketplace.json` | Marketplace manifest — the plugin catalog that Claude Code reads to list available plugins |
| `apps/web/` | Nuxt 4 marketplace website — browsing and discovering plugins |
| `hooks/hooks.json` | Root-level marketplace hooks — SessionStart context loading |

## Module Structure

### `plugins/` — Plugin Output Directory

Contains all plugins distributed through the marketplace. Each plugin has a `.claude-plugin/plugin.json` manifest and optional `skills/`, `commands/`, `hooks/`, and `agents/` directories.

**Important**: Many plugins here are auto-generated by `scripts/cli.ts sync`. Files with a `SYNC.md` marker are vendor-synced and must not be edited manually. Manually-maintained plugins (e.g., `gatekeeper`, `plugin-dev`, `mcp-dev`) do not have `SYNC.md`.

### `external-plugins/` — Type 4 Git Submodules (Gemini CLI Extensions)

Read-only git submodules of Gemini CLI extensions. These are converted to Claude Code plugin format during sync. Contains repos like `nanobanana`, `flutter`, `security`, `firebase`, `chrome-devtools-mcp`.

### `sources/` — Type 1 Git Submodules (Documentation Repos)

Read-only clones of upstream documentation repositories (e.g., `vuejs/docs`, `nuxt/nuxt`, `vitejs/vite`). Skills are generated from these docs using `/generate-skill <name>` and placed directly into `plugins/`.

### `vendor/` — Type 2 & 3 Submodules (Vendor Skills)

- **Type 2**: Repositories that maintain their own `skills/` directory (e.g., `slidev`, `vueuse`, `prisma`, `better-auth`, `ai-sdk`). Skills are copied directly from `vendor/<name>/skills/` to `plugins/<plugin>/skills/` during sync.
- **Type 3**: Hand-written skills by Anthony Fu in `vendor/antfu-skills/skills/`. Copied to `plugins/antfu/skills/`.

### `apps/web/` — Marketplace Website

Nuxt 4 application with Nuxt UI v4, Nuxt Content, and better-sqlite3. Provides a browsable web interface for the plugin marketplace.

## Plugin Type System

The repository manages four plugin types with distinct ingestion pipelines:

```
Type 1: Source Docs → /generate-skill → plugins/{plugin}/skills/
         sources/{name}/  (git submodule of docs repo)

Type 2: Vendor Skills → sync → plugins/{plugin}/skills/
         vendor/{name}/   (git submodule with skills/ dir)

Type 3: Manual Skills → sync → plugins/{plugin}/skills/
         vendor/antfu-skills/skills/  (hand-written)

Type 4: Gemini Extensions → sync → plugins/{name}/
         external-plugins/{name}/  (git submodule)
         Converts gemini-extension.json → plugin.json
         Converts TOML commands → Markdown commands
         Copies context files and generates SessionStart hooks
```

### Sync Data Flow

```
scripts/meta.ts          defines source registrations
        ↓
scripts/cli.ts init      adds git submodules
        ↓
scripts/cli.ts sync      pulls latest + generates artifacts
        ↓
plugins/{name}/          output: plugin.json, skills/, commands/, hooks/
        ↓
marketplace.json         references plugins for user installation
```

## Architecture Invariants

1. **Never edit SYNC.md-marked files**: Files in `plugins/*/skills/` with a `SYNC.md` are overwritten on next `bun run skills:sync`. Edit the upstream source instead.

2. **`hooks/hooks.json` is auto-loaded**: Claude Code auto-discovers `hooks/hooks.json` in plugin directories. Do NOT reference it in `plugin.json`'s `hooks` field — that field is for additional hook files only.

3. **Skills over SessionStart hooks**: New plugins should use `skills/` with `SKILL.md` for intelligent, on-demand context loading. SessionStart hooks waste tokens by loading on every session regardless of relevance.

4. **Submodules are read-only**: `external-plugins/`, `vendor/`, and `sources/` directories contain git submodules. Changes go to upstream repos, not here.

5. **`scripts/meta.ts` is the source of truth**: All plugin source registrations live in `meta.ts`. The `SKILL_TO_PLUGIN` mapping in `cli.ts` determines which plugin directory receives each skill.

6. **Plugin names use kebab-case**: All plugin directory names and `name` fields in `plugin.json` use kebab-case with no spaces.

7. **MCP servers use `npx -y`**: MCP servers published to npm should use `npx -y @org/package` to avoid interactive prompts. Environment variables use `${VAR:-}` for optional values.

8. **Node 22.x required**: The `.nvmrc` specifies Node 22. Bun 1.3.10+ is the package manager.

## Cross-Cutting Concerns

### Testing

- **Framework**: Vitest 4.x with shared config from `@pleaseai/vitest-config`
- **Root config**: `vitest.config.ts` defines project-scoped test roots (`packages/`, `apps/`, `scripts/`)
- **Commands**: `bun run test` (Turborepo), `bun run test:projects` (Vitest direct), `bun run test:integration`
- **Integration tests**: Excluded from default runs, require `--timeout 600000`

### Linting & Formatting

- **ESLint 10** with `@antfu/eslint-config` — config at `eslint.config.mjs`
- **Prettier** for formatting
- **commitlint** with `@commitlint/config-conventional` enforced by Husky

### Build System

- **Turborepo** orchestrates tasks across workspaces (`packages/`, `apps/`, `plugins/`)
- Task graph defined in `turbo.json` — build, lint, test, dev, e2e tasks
- Caching enabled for build, lint, and test tasks

### Security

- `hooks/deny-vendor-write.sh` prevents accidental modification of vendor-synced files
- Path traversal guards in `cli.ts` validate that resolved paths stay within expected directories
- `execFile`/`execFileSafe` used instead of shell execution for external input to prevent command injection
