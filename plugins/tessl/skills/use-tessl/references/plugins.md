# Authoring & Publishing Tessl Plugins

Source: <https://docs.tessl.io/create/creating-plugins> and
<https://docs.tessl.io/create/developing-plugins-locally>.

A **plugin** is a reusable package that codifies team standards and procedural knowledge for coding
agents. It is the unit you publish to the Tessl registry under a `workspace/plugin` namespace.

## What a plugin can contain

| Component | Status | Purpose |
|-----------|--------|---------|
| **Skills** | Available | Procedural, step-by-step agent workflows (API testing, migrations, deploy checklists, reviews) |
| **Rules** | Available | Mandatory, always-loaded coding standards (error handling, validation, naming, security) |
| **Hooks** | Coming soon | Deterministic commands triggered by events (e.g. run a linter after a file edit) |
| **MCP tools** | Coming soon | Custom tools exposing internal APIs/databases/services to agents |

## Directory structure

```
my-plugin/
├── .tessl-plugin/
│   └── plugin.json          # manifest (source of truth)
├── rules/
│   └── standards.md         # always-loaded standards (optional)
└── skills/
    └── my-skill/
        └── SKILL.md          # one skill; add more sibling dirs for more skills
```

## `.tessl-plugin/plugin.json` manifest

Generated by `tessl plugin new` / `tessl skill new`. Key fields:

| Field | Meaning |
|-------|---------|
| `name` | Package name in `workspace/plugin-name` format |
| `version` | Semantic version (semver) |
| `description` | Helper text describing the plugin's purpose |
| `private` | Boolean — visibility scope (private to the workspace vs. public) |
| `skills` | Array of directories that each contain a `SKILL.md` |

> Legacy note: older plugins used a root `tile.json`. Migrate with `tessl tile migrate [--force]`, which
> rewrites it to `.tessl-plugin/plugin.json`.

## Local development loop

The workflow is **create → install from filesystem → test → lint/pack → iterate**:

```bash
# 1. Scaffold the plugin
tessl plugin new --name myworkspace/my-plugin --path ./my-plugin

# 2. Add one or more skills
tessl skill new --name my-skill --path ./my-plugin/skills/my-skill

# 3. Install from the local filesystem to exercise it in a real repo
tessl install file:./my-plugin

# 4. Test:
#    - Skills: invoke them directly and verify the agent executes them correctly.
#    - Rules: ask the agent to generate code and confirm it follows the rule.

# 5. Validate and package
tessl plugin lint ./my-plugin
tessl plugin pack --output ./dist ./my-plugin
```

Re-run `tessl plugin lint` after every change; `file:` installs pick up edits as you iterate.

## Publishing

```bash
tessl login                                # or rely on an API key / setup-tessl in CI
tessl plugin lint ./my-plugin              # never publish something that fails lint
tessl plugin publish --dry-run             # preview the publish
tessl plugin publish --bump minor          # publish + bump (patch | minor | major)
tessl plugin info myworkspace/my-plugin    # confirm what landed in the registry
```

### Retracting / retiring

| Need | Command |
|------|---------|
| Mistaken publish (within 2 days) | `tessl plugin unpublish --plugin myworkspace/my-plugin@1.2.3` |
| Retire but keep existing installs working | `tessl plugin archive --plugin myworkspace/my-plugin --reason "<text>"` |

## Quality gates before publishing

- `tessl plugin lint` — structural validation of the whole plugin.
- `tessl skill lint` / `tessl skill review --optimize` — per-skill validation and AI quality review
  (see [`skills.md`](skills.md)).
- `tessl eval run` against generated `tessl scenario` cases — behavioural evaluation before a major bump.
- In CI, gate PRs with [`tesslio/skill-review`](https://github.com/tesslio/skill-review) and publish on
  push with [`tesslio/setup-tessl`](https://github.com/tesslio/setup-tessl) — see
  [`ci-publishing.md`](ci-publishing.md).
